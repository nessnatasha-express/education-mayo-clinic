02 — What are AEM Forms

Understanding the Platform

An AEM Form is a digital data-capture experience built inside Adobe Experience Manager — it can be a simple contact form or a complex multi-step enrollment flow, rendered on any device, prefilled from backend systems, validated in real time, and connected to downstream workflows for processing and storage. Unlike a standalone form tool, AEM Forms is integrated directly into your content, data, and workflow infrastructure, meaning every form submission becomes a structured event your organization can act on immediately.

4

AEM Forms is not a single form builder. It is a suite of four distinct delivery architectures, each suited to different complexity, team skill, and performance requirements.

Recommended — New Projects

Edge Delivery + Universal Editor

WYSIWYG drag-and-drop authoring. Plain HTML/CSS/JS output. Near-perfect Lighthouse scores. Supports full AEM backend: FDM, Workflows, Adobe Sign, Workfront Fusion. Adobe's default for all new builds in 2025.

For Content Teams

Edge Delivery + Document-Based

Authors work in Microsoft Excel or Google Sheets. No AEM training needed. Data routes directly to spreadsheets. Best for contact forms, surveys, lead-gen, and simple intake — not complex multi-step flows.

Complex Enterprise Forms

Adaptive Forms (Core Components)

30 open-source BEM-compliant components. Same WYSIWYG as AEM Sites. Advanced rules editor, conditional logic, multi-panel navigation. Supports Headless API. Preferred over Foundation Components for all new AEM builds.

Multi-Channel / Omnichannel

Headless Adaptive Forms

AEM provides form logic, state, validation, and submission. Your app provides the UI — React, Angular, native mobile, or chatbot. One form definition renders anywhere. Full design system control.

Legacy — Existing Builds Only

Adaptive Forms (Foundation Components)

Maintenance mode only. No new features being added. If you have existing forms here, maintain them but plan migration to Core Components. Do not start new builds on Foundation Components.

Regulated / Compliance

PDF Forms / HTML5 Forms

XDP template-based forms. Offline fill, barcode scan, data import/export from existing PDFs. Still supported for compliance-heavy use cases requiring PDF output and tamper-proof document trails.

How AEM Forms Processes a Submission

Layer 1

Author

Where forms are designed, templated, and managed. Business users use drag-and-drop or document-based authoring. Developers extend with custom components and FDM connections.

Layer 2

Publish / Edge

Customer-facing delivery. Traditional AEM Publish/Dispatcher for Adaptive Forms, or Edge Delivery CDN for EDS forms. This is where end users fill out and submit the form.

Layer 3

Processing

Behind the firewall. Submitted data runs through AEM Workflows, connects to external systems (CRM, ERP, databases), triggers notifications, and stores compliant records.

04 — Best Practices

What to Do and What to Avoid

Sourced from Adobe's official Experience League documentation and Merkle implementation experience. Applies across form types unless noted.

✓ Do This
  • ✓Use Core Components for all new Adaptive Forms builds
  • ✓Start all new projects with Edge Delivery + Universal Editor
  • ✓Define your Form Data Model before building form fields
  • ✓Upload packages via Forms Manager UI, not CRX Package Manager
  • ✓Create overlays in /app folders — never modify /libs
  • ✓Enable Document of Record for all compliance-sensitive submissions
  • ✓Add CAPTCHA (reCAPTCHA) on all public-facing forms
  • ✓Use fragment-based authoring to reuse common field groups
  • ✓Connect Adobe Analytics to track field-level abandonment
  • ✓Test mobile rendering before every publish
  • ✓Plan templates and custom component requirements before building
  • ✓Target Lighthouse score 100 on all Edge Delivery forms
✕ Avoid This
  • ✕Build new forms on Foundation Components
  • ✕Modify anything in /libs directly
  • ✕Use CRX Package Manager for form package uploads
  • ✕Use document-based authoring for complex multi-panel, multi-step forms
  • ✕Skip server-side validation — client + server both required
  • ✕Assume offline sync works without testing Forms Workspace explicitly
  • ✕Stack sequential storage API calls — batch related data
  • ✕Deploy to production without Lighthouse performance verification
  • ✕Use Foundation Components for new headless API integrations
  • ✕Ignore field-level analytics — abandonment data is actionable
  • ✕Launch forms without testing Adobe Sign integration end-to-end
  • ✕Mix build tools (Vite, Webpack) into Edge Delivery projects

Which Approach Is Right for Your Use Case?

Use this reference table to match your specific scenario to the recommended AEM Forms delivery approach. Each row maps a common Mayo Clinic use case to the right architecture and explains the reasoning.

# Scenario Recommended Approach Why
1 New form, any complexity Edge Delivery + Universal Editor Adobe's 2025 default. Fast delivery, Lighthouse 100, no build pipeline overhead.
2 Non-technical content team needs a form fast Edge Delivery + Document-Based (Excel/Sheets) No AEM training required. Data routes directly to spreadsheets. Live in minutes.
3 Complex form with AEM Workflow, FDM, Adobe Sign Adaptive Forms Core Components or EDS + Universal Editor Full backend capabilities, advanced rules editor, multi-panel flows.
4 Form embedded in React app or native mobile Headless Adaptive Forms One form definition, any UI. React hooks provided. AEM manages all logic.
5 Regulated submission requiring PDF archive Enable Document of Record on Adaptive Forms Auto-generates tamper-proof PDF of every submission. Compliance-ready.
6 Existing Foundation Component forms Maintain; plan migration to Core Components Foundation is not deprecated but receives no new features.
